Xsens Device API Objectives and Considerations

Objectives

The main objective of the SDK is that developers are able to easily implement host applications that use Xsens motion trackers. Further objectives are that the SDK is:
  • easy to understand,
  • easy to use,
  • logical and straightforward,
  • preventing mistakes,
  • future proof,
  • platform independent,
  • useable by a large variety of developers.

Considerations

When supplying an SDK one faces a number of decisions as discussed in the following sections.

Language

The first decision is what programming language to choose. This depends a lot on the market. Xsens technology is often used in industrial and research settings. As such, candidates could for example be: Java, C, C++, Objective-C, Python. An important evaluation is that all these languages can handle a C interface. Therefore an obvious choice is to supply a C interface.
However, C is not an object-oriented language. There are good reasons for choosing object-oriented language when developing complex systems, especially with elaborate GUI and concurrency (This is for example supported by state-of-the art frameworks like Qt). Xsens therefore internally uses C++ to develop software. However, this does not mean that the SDK interface libraries necessarily have to be implemented in C++ as well.
Some other advantages and disadvantages between C and C++ are given in the following table:

C C++
- Manual object-lifetime management + Easy object-lifetime management.
- Difficult to deal with complex objects. + Easily handle complex objects.
- Easy to make mistakes. + Harder to make mistakes.
+ Widely accepted and standardized interface. - Only C++ compatible.
+ Not compiler dependent. - Compiler dependent.
- Platform dependent. - Platform dependent.
Note
platform means combination of OS and processor.

Some other considerations for the SDK are given in the following sections.

Name mangling

Due to name-mangling, C++ libraries are not interchangeable between different compilers. There are possible workarounds to go from one compiler to another, but these all require having both compilers installed on the development PC. This means that, if a C++ interface would be used, it must be supplied for every compiler separately. Although this could be organized, it is preferable if this could be avoided.

Generic types libraries

The SDK must include a generic types-library to handle standard data-types like quaternions, vectors and matrices and the standard operation on these types.

Generic, complexity and flexibility

Another problem is that the SDK should support a wide range of devices and the developer should not need to reinstall or rewrite the code whenever a new type of device is used.

Used internally

It will be preferable to use the SDK also for internal development. For example the MT Manager tool is developed using the API as well. However, the interface should not be cluttered with information that is not used by or relevant for the host application developer of the customer.
